# 금융 지표 대시보드 코드베이스 분석 및 향후 작업 준비

## 1) 현재 구조 요약
- 실행 파일은 `opp.py` 단일 파일 구조입니다.
- UI 레이어(Streamlit)와 데이터 수집 로직(yfinance/FRED/웹 스크래핑)이 같은 파일에 함께 존재합니다.
- 지표별 출처 링크는 `SOURCE_URLS` 딕셔너리로 중앙 관리합니다.

## 2) 기능 흐름(High-level)
1. Streamlit 페이지 기본 설정 및 공통 CSS 적용
2. 데이터 수집 함수 실행
   - `get_yf_data`: 야후 파이낸스 티커 기반 종가/전일대비
   - `get_fred_data`: FRED 시계열 기반 최신값/전일대비
   - `get_krx_gold`: 네이버 금융 DOM 파싱
   - `get_fear_and_greed`: CNN API JSON 파싱
3. `render_metric_with_link`로 지표 타이틀+출처 링크+메트릭 표시
4. 섹션별 레이아웃(증시/환율/국채/금/심리/유가) 렌더링

## 3) 현재 코드의 장점
- 단일 진입점이라 배포/실행이 단순합니다.
- 주요 외부 소스가 함수 단위로 분리되어 있어 지표 추가 시 재사용이 쉽습니다.
- 실패 시 `0, 0` 또는 "조회불가"로 안전하게 폴백하여 화면 중단을 방지합니다.

## 4) 리스크 및 개선 우선순위

### P0 (요구사항 대응 전 필수 정비)
1. **도메인 분리**
   - `opp.py`를 `data_sources.py`, `formatters.py`, `ui_sections.py`, `app.py` 등으로 분리
   - 목적: 변경 영향 범위 축소, 테스트 가능성 확보
2. **에러 가시성 확보**
   - 현재는 broad `except`로 오류 원인이 묻힘
   - 최소한 로깅(예: `logging.warning`) 추가하여 운영 이슈 추적 가능하게 개선
3. **데이터 정합성 규칙 명시**
   - 채권 수익률 단위(%, bp), 금/환율 단위 표시 규칙 문서화

### P1 (품질/성능)
1. **캐싱 도입 (`st.cache_data`)**
   - 동일 요청 반복 호출 절감 (특히 페이지 리로드 시)
2. **타임아웃/재시도 정책 표준화**
   - requests 호출에 대해 공통 세션과 재시도 정책 도입
3. **지표 메타데이터 테이블화**
   - 티커, 소스, 포맷 문자열을 구조화하여 반복 코드를 축소

### P2 (확장성)
1. **테스트 기반 마련**
   - 순수 함수(포맷팅/계산)부터 단위테스트 추가
2. **환경 분리**
   - `.env` 또는 설정 파일로 기본 옵션(타임아웃, lookback 기간 등) 분리
3. **관측성**
   - 실패율, 응답시간 등을 간단히 기록해 데이터 소스 신뢰도 관리

## 5) 향후 요구사항 작업을 위한 즉시 실행 체크리스트
- [ ] 모듈 분리 브랜치 생성 후 파일 구조 정리
- [ ] 지표별 스펙 문서(이름/단위/소스/갱신주기/실패 시 처리) 작성
- [ ] 공통 유틸(숫자 포맷, 델타 포맷, 링크 렌더 함수) 분리
- [ ] 네트워크 호출 함수에 로깅 + 타임아웃 + 재시도 적용
- [ ] 최소 단위 테스트 3개(금 프리미엄 계산, 델타 표기, 폴백 동작) 추가

## 6) 빠른 다음 단계 제안
1. 1차 리팩터링: 파일 분리 + 동작 동일성 유지
2. 2차 안정화: 캐싱/로깅/재시도
3. 3차 기능 확장: 신규 지표 추가(예: 비트코인, 달러인덱스, VIX)
